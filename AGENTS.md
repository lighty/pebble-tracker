# Repository Guidelines

## エージェント向け指示
このリポジトリに関する議論、レビュー、実装メモは原則として日本語で行ってください。外部仕様、API 名、識別子などで英語が必要な場合のみ英語を併用します。

## プロジェクト構成
このリポジトリは現在、Obsidian プラグインの最小実装を含みます。役割ごとに以下のように整理します。

- `main.js`: プラグイン本体
- `manifest.json`: Obsidian プラグイン定義
- `styles.css`: UI スタイル
- `docs/`: 設計メモやアーキテクチャ文書
- `src/`: 将来 TypeScript へ移行する場合のアプリケーションコード置き場
- `test/` または `tests/`: 自動テスト
- `assets/`: 画像や固定データなどの静的ファイル

ファイル名は機能単位で分かるように付けます。例: `src/tracker.ts`、`tests/tracker.test.ts`

## ビルド・テスト・開発コマンド
まだビルドツールやテストランナーは導入していません。現時点の主要確認コマンドは以下です。

- `ls -la`: ワークスペースの確認
- `find . -maxdepth 2`: ディレクトリ構成の確認
- `node --check main.js`: プラグイン本体の構文確認

新しいツールチェーンを導入する場合は、`Makefile` や package scripts など、共通の入口を用意し、`README.md` とこのファイルの両方を更新してください。

## コーディング規約と命名
Markdown、JSON、YAML、フロントエンドコードは原則 2 スペースでインデントします。ただし、採用言語に強い慣習がある場合はその規約を優先してください。

- モジュールは小さく保ち、責務を 1 つに絞る
- 省略形より意味の明確な名前を使う
- 生成物は必要な場合だけコミットする
- 最初の実装時に formatter と linter を追加する

## テスト方針
まだテストフレームワークは設定されていません。最初の本番コードを追加する時点で、自動テストも合わせて導入してください。テスト名は採用技術に合わせて統一します。例: `tracker.test.ts`、`tracker_spec.rb`

新機能や不具合修正には、可能な限り最低 1 件の自動テストを追加してください。テストコマンドが決まったら、この文書にも追記します。

## コミットとプルリクエスト
このディレクトリにはまだ Git 履歴がないため、コミットメッセージは命令形で簡潔に書いてください。例: `Add initial tracker model`、`Set up test harness`

プルリクエストでは次を満たしてください。

- 変更内容と目的を簡潔に説明する
- 関連 Issue があればリンクする
- UI 変更がある場合だけスクリーンショットを付ける
- 未対応事項や今後の課題があれば明記する
